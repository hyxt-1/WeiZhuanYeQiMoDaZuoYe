# Todo List 待办事项系统 - 开发过程文档

## 一、项目概述

本项目是一个全栈待办事项（Todo List）管理系统，采用前后端分离架构实现。用户可以登录系统，创建、编辑、删除任务，并通过看板视图（待完成/进行中/已完成三列）管理任务的状态流转。

### 技术栈

**后端：**
- 框架：FastAPI 0.115.0
- ORM：SQLAlchemy 2.0.35
- 数据库：SQLite
- 认证：JWT (python-jose) + 密码哈希 (passlib + pbkdf2_sha256)
- 数据验证：Pydantic 2.9.0
- ASGI 服务器：Uvicorn 0.32.0

**前端：**
- 框架：Vue 3.4 + TypeScript 5.4
- 构建工具：Vite 5.4
- 路由：Vue Router 4.3
- HTTP 客户端：Axios 1.6
- UI 样式：TailwindCSS 3.4

## 三、开发步骤

### 阶段一：需求分析与技术选型

**需求梳理：**
1. 用户登录/登出，基于 Token 的身份认证
2. 任务的增删改查（CRUD）
3. 任务状态流转：待完成 → 进行中 → 已完成（支持回退）
4. 任务属性：标题、描述、优先级（高/中/低）、标签、截止时间
5. 任务搜索（按标题、描述、标签）
6. 看板视图展示（三列拖拽式布局的简化版，使用按钮切换状态）

**技术选型理由：**
- **FastAPI**：自动生成 OpenAPI 文档、类型安全、异步支持、开发效率高
- **SQLite + SQLAlchemy**：零配置数据库，ORM 抽象便于后续迁移到 MySQL/PostgreSQL
- **JWT 认证**：无状态认证，适合前后端分离架构
- **Vue 3 + TS**：Composition API 组织逻辑更清晰，TypeScript 减少运行时错误
- **TailwindCSS**：原子化 CSS，快速构建美观 UI，无需编写大量样式文件
- **Vite**：开发环境冷启动快，HMR 响应即时

---

### 阶段二：后端开发

#### Step 1：数据库层搭建

1. 编写 `database.py`：配置 SQLite 连接，创建 `SessionLocal` 和 `Base` 基类，实现 `get_db()` 依赖注入生成器（确保请求结束后关闭连接）。

2. 编写 `models.py`：定义 `User` 和 `Task` 两个 ORM 模型。
   - `User` 表：id、username（唯一）、hashed_password、email（唯一）、created_at
   - `Task` 表：id、title、description、status（todo/in_progress/completed）、priority（low/medium/high）、tags（JSON 数组）、due_date、created_at、updated_at

3. 编写 `schemas.py`：使用 Pydantic 定义请求/响应模型。
   - 区分 `TaskBase` / `TaskCreate` / `TaskUpdate`：`TaskUpdate` 全部字段可选，支持部分更新
   - `User` Schema 不暴露 `hashed_password` 字段
   - 配置 `from_attributes = True`（Pydantic v2 的 `orm_mode` 替代品）支持 ORM 对象直接序列化

#### Step 2：认证模块实现

编写 `auth.py`，核心逻辑包括：

1. **密码哈希**：使用 `passlib` 的 `CryptContext`，算法选择 `pbkdf2_sha256`。
   > *注意*：最初打算使用 bcrypt，但 bcrypt 在 Windows + Python 3.13 环境下编译依赖容易出错，pbkdf2_sha256 是 passlib 内置算法，兼容性更好，安全性同样满足要求。

2. **JWT Token 生成与验证**：
   - 使用 `python-jose` 的 `jwt.encode/decode`
   - 算法 HS256，过期时间 30 分钟
   - Payload 中存放 `sub`（用户名）和 `exp`（过期时间戳）

3. **依赖注入 `get_current_user`**：
   - 通过 `OAuth2PasswordBearer` 从请求头提取 `Authorization: Bearer <token>`
   - 解析 JWT → 查库获取 User 对象 → 返回给路由函数
   - 解析失败或用户不存在时抛出 401

#### Step 3：主应用与路由

编写 `app.py`：

1. 启动时调用 `Base.metadata.create_all(bind=engine)` 自动建表
2. 配置 CORS 中间件（`allow_origins=["*"]`，开发阶段方便调试，生产环境应限制具体域名）
3. 挂载 `/api/auth/login` 登录路由：验证用户名密码 → 生成 Token → 返回 `{access_token, token_type, user}`
4. 挂载任务路由（`tasks.py`）到 `/api/tasks` 前缀

编写 `tasks.py`（任务 CRUD）：

| 方法   | 路径           | 功能             | 认证 |
|--------|----------------|------------------|------|
| GET    | /api/tasks     | 任务列表+搜索    | 是   |
| GET    | /api/tasks/{id}| 单个任务详情     | 是   |
| POST   | /api/tasks     | 创建任务         | 是   |
| PUT    | /api/tasks/{id}| 更新任务         | 是   |
| DELETE | /api/tasks/{id}| 删除任务         | 是   |

- 搜索：`?search=关键词` 在 title 和 description 上做 `contains` 模糊查询
- 更新：使用 `task.dict(exclude_unset=True)` 只更新传入的字段，未传字段保持原值

#### Step 4：初始化脚本

编写 `init_db.py`：建表 + 检查是否存在 admin 用户，不存在则插入默认账号 `admin / admin123`。

## 四、遇到的问题与解决方案

### 问题 1：Windows 下 bcrypt 依赖安装失败

**现象：**
`passlib[bcrypt]` 在 Python 3.13 + Windows 环境下安装时，`bcrypt` 需要编译 C 扩展，报错 `error: Microsoft Visual C++ 14.0 or greater is required`。

**解决方案：**
将密码哈希算法从 `bcrypt` 切换为 passlib 内置的 `pbkdf2_sha256`。该算法基于 SHA-256 的 PBKDF2，同样是 NIST 推荐的密码存储方案，且不需要任何外部编译依赖，跨平台兼容性极佳。对应修改 `auth.py` 第 15 行：
```python
pwd_context = CryptContext(schemes=["pbkdf2_sha256"], deprecated="auto")
```
同时 `requirements.txt` 中的 `passlib[bcrypt]` 保留不影响，实际只用到内置算法。

---

### 问题 2：前后端跨域（CORS）

**现象：**
前端 Vite 开发服务器运行在 `http://localhost:5173`，后端 FastAPI 在 `http://localhost:8000`，浏览器出于同源策略阻止 Axios 请求，控制台报错 `CORS policy: No 'Access-Control-Allow-Origin' header`。

**解决方案（双保险）：**

1. **后端加 CORS 中间件**（`app.py` 第 15-21 行）：`CORSMiddleware` 放行所有来源、方法、请求头。开发阶段使用 `allow_origins=["*"]`，部署到生产环境时改为具体前端域名。

2. **前端 Vite 开发代理**（`vite.config.ts` 第 13-17 行）：前端请求 `/api/*` 路径时，Vite 自动代理转发到 `http://localhost:8000`。这样浏览器视角下前后端同源，完全绕过 CORS，同时方便打包部署时替换为真实 API 域名。

---

### 问题 3：Pydantic v2 与 v1 API 不兼容

**现象：**
项目使用 Pydantic 2.9.0，v1 时代常见的 `orm_mode = True` 在 v2 中已被重命名，按老写法会报 `ValueError: "orm_mode" is removed in Pydantic v2`。

**解决方案：**
将 Schema 内部配置改为 `from_attributes = True`（Pydantic v2 新命名）：
```python
class Config:
    from_attributes = True
```
同时 ORM 模型转 Schema 的 API 也从 `from_orm()` 改为 `model_validate()`，但 FastAPI 内部 `response_model` 已经做了适配，只需配置 Config 即可。

---

### 问题 4：TaskList.vue 状态切换函数引用未定义

**现象：**
`TaskList.vue` 第 349-356 行 `changeTaskStatus` 函数中调用了 `updateTask` API，但是 script 顶部的 import 语句漏掉了 `updateTask`，运行时 `Uncaught ReferenceError: updateTask is not defined`。

**解决方案：**
将第 267 行的 import 从：
```ts
import { getTasks, createTask, deleteTask } from '../api'
```
修正为：
```ts
import { getTasks, createTask, deleteTask, updateTask } from '../api'
```
这是开发时常见的"写了函数调用但忘了补 import"的低级错误，TypeScript 在 `<script setup>` 中会报红线提示，严格模式下会拦截构建。

---

### 问题 5：SQLite 多线程访问报错

**现象：**
FastAPI 请求处理跑在多线程下，默认 SQLite 连接不允许跨线程使用，报错 `SQLite objects created in a thread can only be used in that same thread`。

**解决方案：**
`database.py` 第 8 行创建 engine 时传入连接参数：
```python
connect_args={"check_same_thread": False}
```
明确告诉 SQLite 允许跨线程共享连接。配合 `get_db()` 的依赖注入机制（每个请求独立 Session，结束即 close），在并发量不大的场景下工作稳定。如果后续迁移到 PostgreSQL 就完全不需要这个参数。

---

### 问题 6：前端日期字段前后端格式不一致

**现象：**
新建任务时 `<input type="date">` 返回的 `due_date` 是 `YYYY-MM-DD` 字符串，而后端 Pydantic Schema 中 `due_date` 类型是 `Optional[datetime]`，传空字符串 `""` 时会触发 `Input should be a valid datetime` 校验错误。

**解决方案：**
1. 后端 `schemas.py` 中 `TaskBase.due_date` 已声明为 `Optional[datetime] = None`，只要前端不传空字符串而传 `null`/`undefined` 即可
2. 前端 API 封装时 Axios 会自动跳过值为空字符串的字段？实际需要在新建任务逻辑里做一次清洗：`due_date` 为空字符串时 delete 该字段。目前后端默认值为 `None`，在当前简单场景下只有用户手动选了日期才会传值，因此实际未触发 bug，属于待加固项。

---

### 问题 7：登录接口 OAuth2 与自定义 Schema 冲突

**现象：**
`auth.py` 中 `oauth2_scheme = OAuth2PasswordBearer(tokenUrl="auth/login")` 会自动把 `tokenUrl` 作为 OpenAPI 文档的授权地址，OAuth2PasswordBearer 默认期望登录接口返回 Form 表单（`application/x-www-form-urlencoded`）并返回 `access_token` 字段。但前端登录用的是 JSON body。

**解决方案：**
登录接口不使用 FastAPI 的 `OAuth2PasswordRequestForm`，而是自己写 `LoginRequest` Schema 接收 JSON。`OAuth2PasswordBearer` 仅用来作为依赖从 Header 取 Token，不参与登录流程。这样前后端都用 JSON，逻辑更一致；Swagger UI 的"Authorize"按钮可以手动贴 Token 调试。

## 六、后续可优化方向

1. **注册功能**：目前只有 init_db.py 创建的硬编码 admin 用户，可增加 `/api/auth/register` 注册接口和前端注册页
2. **任务归属**：Task 表缺少 `user_id` 外键，当前所有用户共享一份任务列表；加外键后每个用户只能看自己的任务
3. **分页**：任务量大后列表需要分页加载，后端加 `offset/limit` 参数
4. **拖拽排序**：三列看板可引入 `vuedraggable` 实现拖拽改变状态和顺序
5. **数据库迁移**：引入 Alembic 管理表结构变更，替代 `create_all` 自动建表
6. **密钥管理**：`auth.py` 中 `SECRET_KEY` 硬编码，应改为从环境变量读取（`python-dotenv`）
7. **CORS 收紧**：`allow_origins=["*"]` 改为具体域名白名单
8. **单元测试**：后端 pytest 覆盖 CRUD + 认证，前端 Vitest 覆盖组件逻辑
9. **截止时间提醒**：超过 due_date 且状态非 completed 的任务高亮或红点提醒
